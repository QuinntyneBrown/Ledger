# L1 - High-Level Requirements

**Product:** Ledger — a ruthlessly simple weight tracker. Set a goal, log a weigh-in, read the trend, celebrate milestones. Dark-first, premium, calm.

**Delivery:** Ledger is a **responsive web application** — a single Angular SPA served over the browser, not a native mobile app. The reference mockups are mobile-first (a phone frame with a bottom tab bar); the shipped app must render from that single-column mobile layout up through tablet and desktop, adapting layout and navigation at every breakpoint (XS < 576px, SM ≥ 576px, MD ≥ 768px, LG ≥ 992px, XL ≥ 1200px).

**Stack & conventions:** Identical to [Liturgy](https://github.com/QuinntyneBrown/Liturgy) — a .NET 9 / ASP.NET Core backend using CQRS with MediatR, EF Core over SQL Server, JWT bearer authentication, and SignalR for real-time updates; an Angular 21 / TypeScript / RxJS frontend organized as an `app` shell plus `api`, `components`, and `domain` libraries; xUnit + `WebApplicationFactory` integration tests on the backend and Jest + Playwright on the frontend. Clean-architecture layering (`Domain`, `Application`, `Infrastructure`, `Api`) with test projects mirroring source.

> These are **production-grade** requirements: security-first, high-performance, responsive (XS through XL), and driven by Acceptance Test Driven Development (ATDD). Every acceptance test written during implementation must carry a `Traces to: L2-xxx` comment header.

---

## L1-001: Account Registration & Authentication
The system must let a person create an account, verify their email, sign in, sign out, recover a forgotten password, and change their password. Authentication must be stateless (JWT bearer) and protect credentials at rest and in transit.

## L1-002: Onboarding & Initial Goal Setup
The system must guide a newly registered user through a short, resumable onboarding flow that captures their weight unit, current weight, goal weight, and target date, and establishes their starting baseline before they reach the main app.

## L1-003: Weight Logging (Weigh-ins)
The system must let an authenticated user record a weight for a given day — for today via a quick action available anywhere, or for a past day via a date picker — adjust it in fine increments, attach an optional note, and edit or delete any recorded entry.

## L1-004: Goal Management & Progress Projection
The system must track the user's goal (start weight, goal weight, target date), compute progress toward it, project a completion date from recent pace, indicate whether the user is ahead of, on, or behind schedule, and let the user change the goal.

## L1-005: Trends & Analytics
The system must visualize weight over time — daily values, a smoothed moving average, a range band, and the goal line — across selectable time ranges, and surface summary analytics (weekly rate, total change, best week, BMI) derived from the user's entries.

## L1-006: Weigh-in History Management
The system must present the full history of a user's entries grouped by month with per-month change, allow sorting and filtering, and allow editing or deleting any entry, remaining performant for users with thousands of entries.

## L1-007: Milestones, Badges & Streaks
The system must award badges for defined achievements (first entry, weight-loss thresholds, logging streaks, entry-count and goal milestones), track current and longest logging streaks, show progress toward the next badge, and celebrate unlocks.

## L1-008: Dashboard (Home)
The system must provide a home screen that summarizes the user's current standing at a glance — progress dial, weekly and streak stats, a recent-trend snapshot, and the next milestone — and must present coherent empty, loading, and celebration states.

## L1-009: User Profile
The system must maintain a user profile (name, email, height, avatar, membership date) used for display and for BMI calculation, and let the user view and edit it, re-verifying email on change.

## L1-010: Preferences & Settings
The system must let the user configure display and behavior preferences — weight unit (kg/lbs), theme (light/dark/system), and the day the week starts on — and persist them across devices and sessions.

## L1-011: Reminders & Notifications
The system must let the user enable a daily weigh-in reminder at a chosen time, define quiet hours, and must suppress the reminder on days a weight has already been logged.

## L1-012: Data Export & Account/Data Deletion
The system must let the user export all of their entries as CSV, permanently delete all of their tracking data behind an explicit typed confirmation, and delete their account entirely, honoring data-protection obligations.

## L1-013: Real-Time Multi-Device Synchronization
The system must keep a user's data consistent across their signed-in devices, pushing new, edited, and deleted entries and goal changes to other active sessions in real time via SignalR.

## L1-014: Marketing Website
The system must include a public, responsive marketing website that communicates Ledger's value proposition, showcases its features, and converts visitors into registered users, independent of the authenticated application.

## L1-015: Design System (Aurora) & Responsive Web UI
Ledger must be delivered as a responsive web application built on the Aurora design system (tokens, foundations, and components). Every screen must render correctly in both light and dark themes and adapt fluidly across extra-small, small, medium, large, and extra-large viewports — from the mobile-first single-column layout with a bottom tab bar up to a multi-column desktop layout with persistent side navigation — using a fluid, touch- and pointer-friendly interface with no horizontal scrolling at any supported width.

## L1-016: Security & Data Protection
The system must enforce that users can access only their own data, validate and encode all input and output, defend against the OWASP Top 10, secure transport and secrets, rate-limit abusable endpoints, and audit security-relevant events.

## L1-017: Performance & Scalability
The system must meet defined response-time, rendering, and throughput targets, compute trend and aggregate analytics efficiently, and scale to a growing user base without degradation.

## L1-018: Accessibility
The application and marketing site must meet WCAG 2.1 AA, support full keyboard operation with correct focus management, expose correct screen-reader semantics, meet contrast requirements, and respect reduced-motion preferences.

## L1-019: Observability & Operations
The system must emit structured logs with correlation identifiers, expose metrics and health checks, track and alert on errors, and support safe operation and diagnosis in production.

## L1-020: Architecture, Code Organization & Conventions
The codebase must mirror the Liturgy reference architecture and conventions: clean-architecture CQRS backend with MediatR, EF Core migrations over SQL Server, a versioned REST API with a consistent error contract, an Angular workspace split into `app`/`api`/`components`/`domain`, and the prescribed testing stack with ATDD traceability.
