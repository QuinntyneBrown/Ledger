Forms & validation

Ledger asks for very little — a weight, a goal, a date. Every field should feel like a quiet, confident conversation. This guide covers how to compose a field, when to validate, and how to surface errors so a form stays calm and forgiving rather than scolding.

Anatomy of a field

A field is three parts stacked in a single .au-field column: a label, the control, and help text. The help slot is permanent — it holds a hint by default and swaps to an error message in place, so the layout never jumps.

When to validate

Timing is the difference between a form that helps and one that nags. Aurora forms follow one rule: never show an error for a field the person is still working on.

  • Validate on blur, once the field is complete and focus has moved on — not on every keystroke.
  • Once a field is in error, forgive on input: re-check as they type and clear the error the moment it's valid, so the correction feels instant.
  • Never block typing. Constrain with inputmode and accept-then-validate; don't reject characters mid-entry.
  • Save the full sweep — and the error summary — for the submit attempt.
  • Use success styling sparingly. A green tick on every field is noise; reserve it for values worth confirming, like a matched password or an available handle.
The forgiveness curve. Validate late and clear early. A person mistypes their goal, tabs away, sees the coral hint, fixes one digit — and the error vanishes before they finish. Errors that appear at the first keystroke train people to distrust the form.

Required markers

Mark required fields, not optional ones — in a short form most fields are required, so flagging the exceptions is quieter. Use the coral asterisk from .au-req and pair it with an accessible cue: give the input required aria-required="true" so it isn't colour-alone.

When only one or two fields are optional, append a muted “Optional” to their label instead of asterisking everything else.

Error summary

When a submit fails, lead with a single summary at the top of the form. It confirms what went wrong, counts the problems, and links straight to each field. Give it role="alert" and move focus to it so screen readers announce the failure once.

Accessibility. Tie help and error text to the input with aria-describedby, set aria-invalid="true" on failure, and never signal state with colour alone — every error pairs the coral with an alert-circle icon and a sentence.
