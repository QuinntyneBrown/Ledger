# L2 - Detailed Requirements

Each L2 traces to exactly one L1. Acceptance criteria use Given/When/Then so they translate directly into tests. During implementation, every acceptance test must carry a `Traces to: L2-xxx` comment header.

**Conventions used below**
- "Weight value" means a number stored canonically in kilograms with one decimal place; the display unit (kg or lbs) is a per-user preference (L2-045). Adjustment steps are 0.1 in the active unit.
- "Owner" means the authenticated user whose id matches the resource's `userId`.
- Responsive breakpoints: **XS** < 576px, **SM** ≥ 576px, **MD** ≥ 768px, **LG** ≥ 992px, **XL** ≥ 1200px. Unless stated otherwise, every UI-facing L2 must satisfy: **Given** any supported viewport width, **when** the screen renders, **then** no horizontal page scrolling occurs and all interactive targets are ≥ 44×44px on touch pointers.

## L2-001: User registration
**Traces to:** L1-001

A visitor creates an account with full name, email, and password, and must accept the Terms of Service and Privacy Policy.

**Acceptance Criteria:**
1. Given valid name, a unique well-formed email, a password meeting policy (L2-002), and terms accepted, when the user submits registration, then an account is created in the unverified state and a `201` with no password material is returned.
2. Given an email already registered, when the user submits, then registration is rejected with a field-level error "That email is already in use" and a link to sign in, and no account is created.
3. Given the terms checkbox is unchecked, when the user attempts to submit, then submission is blocked and the terms requirement is indicated.
4. Given a malformed email or missing required field, when the user submits, then per-field validation errors are shown and no request reaches persistence beyond validation.
5. Given any registration attempt, when the response is returned, then it never reveals whether an email exists via timing or differing error shape beyond the explicit "already in use" case for authenticated-safe UX (see L2-071 rate limiting).
6. Given viewport widths XS through XL, when the sign-up screen renders, then the form is single-column and legible with inputs at full container width on XS and constrained to a centered card (max ~480px) on MD and above.

## L2-002: Password policy & secure storage
**Traces to:** L1-001

Passwords must meet a strength policy and be stored using a slow, salted one-way hash.

**Acceptance Criteria:**
1. Given a password shorter than 8 characters, or lacking at least one number and one symbol, when submitted, then it is rejected with guidance and a live strength meter reflects strength as the user types.
2. Given a valid password, when the account is persisted, then only a salted hash from an approved algorithm (bcrypt/argon2/PBKDF2 with per-user salt and cost factor) is stored — never the plaintext or a reversible form.
3. Given a stored password hash, when any API response or log is produced, then no password or hash value is ever included.
4. Given a password field, when rendered, then a show/hide toggle is available and defaults to hidden.

## L2-003: Email verification
**Traces to:** L1-001

New accounts must verify their email before the account is considered fully active.

**Acceptance Criteria:**
1. Given a newly registered account, when registration completes, then a verification email with a single-use, time-limited token (≤ 24h) is sent.
2. Given a valid unexpired token, when the verify-email link is opened, then the account is marked verified and the user is directed onward.
3. Given an expired or already-used token, when opened, then verification fails with a clear message and an option to resend.
4. Given an unverified account, when it requests a new verification email, then requests are rate-limited (L2-071) and previous tokens are invalidated.

---

## L2-004: Login & JWT issuance
**Traces to:** L1-001

Registered users sign in with email and password to receive a JWT access token.

**Acceptance Criteria:**
1. Given correct credentials for a verified account, when the user signs in, then a signed JWT access token (short-lived) and a refresh token are issued and the user reaches the dashboard.
2. Given incorrect credentials, when the user signs in, then a generic "email or password is incorrect" error is returned without disclosing which was wrong.
3. Given repeated failed attempts beyond the threshold, when the user tries again, then further attempts are throttled/locked per L2-009.
4. Given a successful login, when the token is issued, then it contains the user id and expiry and is signed with a server-held secret; it must not contain sensitive PII beyond the user identifier.

## L2-005: Forgot-password request
**Traces to:** L1-001

A user who forgot their password can request a reset link.

**Acceptance Criteria:**
1. Given any submitted email, when a reset is requested, then the same neutral confirmation ("If that email exists, we've sent a link") is shown regardless of whether the account exists.
2. Given an existing account, when a reset is requested, then a single-use, time-limited (≤ 1h) reset token is emailed and any prior reset token is invalidated.
3. Given repeated requests, when submitted, then they are rate-limited per L2-071.

## L2-006: Reset password
**Traces to:** L1-001

A user completes a password reset using the emailed token.

**Acceptance Criteria:**
1. Given a valid unexpired reset token and a new password meeting policy (L2-002), when submitted, then the password is updated, all existing sessions/refresh tokens are revoked, and the user is directed to sign in.
2. Given an expired, used, or invalid token, when submitted, then the reset is rejected with an option to request a new link.
3. Given a successful reset, when it completes, then a confirmation notification is sent to the account email.

---

## L2-007: Change password (authenticated)
**Traces to:** L1-001

A signed-in user changes their password by confirming the current one.

**Acceptance Criteria:**
1. Given the correct current password and a new password meeting policy, when submitted, then the password is updated and other active sessions are revoked while the current session remains valid.
2. Given an incorrect current password, when submitted, then the change is rejected without altering credentials.
3. Given a successful change, when it completes, then the "last updated" timestamp shown in the profile/security area reflects the new time.

---

## L2-008: Session management & sign-out
**Traces to:** L1-001

Sessions must be renewable and revocable.

**Acceptance Criteria:**
1. Given a valid refresh token, when the access token expires, then a new access token is issued without re-entering credentials.
2. Given the user chooses "Sign out", when confirmed, then the current session's tokens are revoked, client-side tokens are cleared, and the user is returned to the login screen while their stored data is retained.
3. Given a revoked or expired token, when used against any protected endpoint, then the request is rejected with `401`.
4. Given tokens are stored client-side, when persisted, then they are kept in a way not readable by third-party script (e.g. httpOnly refresh cookie) and are never placed in URL query strings.

---

## L2-009: Authentication throttling & lockout
**Traces to:** L1-001

Repeated failed authentication must be slowed and locked.

**Acceptance Criteria:**
1. Given N consecutive failed logins (configurable, default 5) for an account or source IP, when the next attempt is made, then it is rejected with a back-off and the event is audit-logged (L2-072).
2. Given a temporary lockout window has elapsed, when the user retries with correct credentials, then login succeeds.
3. Given lockout is active, when responses are returned, then they do not reveal account existence.

## L2-010: Onboarding wizard (units, current weight, goal, target date)
**Traces to:** L1-002

After first sign-in, the user completes a 4-step wizard with a visible step indicator (e.g. "2/4").

**Acceptance Criteria:**
1. Given a freshly verified account with no goal, when the user enters the app, then the onboarding welcome is shown followed by the four steps: (1) unit selection kg/lbs, (2) current weight, (3) goal weight, (4) target date.
2. Given each step, when the user advances, then a progress indicator updates and the user can navigate Back without losing earlier entries.
3. Given the current-weight step, when the user adjusts the value, then it changes in 0.1 steps in the selected unit and is bounded per L2-015.
4. Given the target-date step, when the user picks a date, then only dates strictly after today are selectable.
5. Given the goal-weight step, when a goal equal to current weight is entered, then the user is warned and cannot complete until goal ≠ current.
6. Given all four steps completed, when the user finishes, then the wizard commits (L2-011) and routes to the dashboard.
7. Given viewport widths XS through XL, when the wizard renders, then steps remain single-column and vertically centered, with the action row pinned to the bottom on XS and inline on MD+.

---

## L2-011: Onboarding commits baseline goal and starting entry
**Traces to:** L1-002

Completing onboarding establishes the user's baseline.

**Acceptance Criteria:**
1. Given completed wizard inputs, when onboarding finishes, then a Goal is created with `startWeight` = entered current weight, `goalWeight`, `targetDate`, and `createdAt` = now.
2. Given onboarding finishes, when the baseline is set, then a first weigh-in entry dated today is created from the current weight, seeding the trend.
3. Given the goal is created, when the dashboard first loads, then progress is 0% and "start" equals "current".

---

## L2-012: Onboarding guard & resumability
**Traces to:** L1-002

Users cannot reach the main app without an established goal, and can resume an interrupted wizard.

**Acceptance Criteria:**
1. Given a user with no goal, when they navigate to any main app route, then they are redirected into onboarding.
2. Given a user abandoned onboarding mid-way, when they return, then the wizard resumes at the last incomplete step with prior entries preserved.
3. Given a user with a completed goal, when they attempt to open onboarding, then they are redirected to the dashboard.

## L2-013: Log today's weight
**Traces to:** L1-003

The user records today's weight with a stepper and optional note.

**Acceptance Criteria:**
1. Given the log-weight screen, when the user taps −/+, then the value changes by 0.1 in the active unit; press-and-hold accelerates repeat adjustment.
2. Given a weight value and optional note (≤ 280 chars), when the user saves, then an entry for today is persisted for the owner and a success toast "Weight logged — {value} {unit}" with an Undo action is shown.
3. Given a note containing markup, when saved and later displayed, then it is stored and rendered as text without executing any embedded markup (L2-068).
4. Given the screen shows context, when a today value is set, then the delta vs the most recent prior entry and the distance to goal are displayed.
5. Given the entry is saved, when persisted, then trends, dashboard, streaks, and badges reflect it without a manual refresh (L2-055).
6. Given viewport widths XS through XL, when the screen renders, then the numeric field and stepper remain centered and touch-friendly, expanding to a constrained centered column on MD+.

## L2-014: Quick weigh-in from anywhere
**Traces to:** L1-003

A floating action control opens a quick weigh-in sheet from any primary screen.

**Acceptance Criteria:**
1. Given any primary screen (dashboard, trends, goal, badges, settings), when the user activates the "+" action, then a bottom sheet dialog for today's weight opens with `role="dialog"` and `aria-modal="true"`.
2. Given the sheet is open, when the user logs the value, then the entry is saved (as L2-013), the sheet closes, and a toast with Undo is shown.
3. Given the sheet is open, when the user presses Escape or the close control or the scrim, then it dismisses without saving.
4. Given the sheet opens, when focus moves, then focus is trapped within the sheet and returns to the invoking control on close (L2-079).

## L2-015: Weight value validation & bounds
**Traces to:** L1-003

Weight inputs must be validated on client and server.

**Acceptance Criteria:**
1. Given a weight below 20.0 kg or above 500.0 kg (equivalent in lbs), when submitted, then it is rejected with a clear bound message on both client and server.
2. Given a value with more than one decimal place, when submitted, then it is rounded/validated to a single decimal in canonical kg.
3. Given a non-numeric or empty value, when submitted, then the save action is disabled/blocked and no request is persisted.
4. Given a value entered in lbs, when persisted, then it is converted to canonical kg without cumulative rounding error across unit toggles.

## L2-016: One entry per calendar day (upsert)
**Traces to:** L1-003

At most one weight entry exists per user per calendar day.

**Acceptance Criteria:**
1. Given an entry already exists for a date, when the user logs another for that same date, then the existing entry is updated (not duplicated) and the change is reflected everywhere.
2. Given "today" is determined, when an entry is dated, then the user's local calendar day (per their timezone) governs uniqueness.
3. Given a concurrent double-submit for the same day, when both reach the server, then exactly one entry results (idempotent upsert enforced by a unique constraint on `userId+date`).

## L2-017: Backdate a past day
**Traces to:** L1-003

The user logs a weight for a previous day via a calendar picker.

**Acceptance Criteria:**
1. Given the backdate calendar, when it renders, then today is marked, past days are selectable, and future days are visually muted and non-interactive.
2. Given a future date, when the user attempts to select or submit it, then it is blocked on client and rejected on server with "Future dates can't be logged".
3. Given a selected past date and weight, when saved, then the entry is inserted/updated for that date, history and trend re-average, and a toast "Saved {value} for {date}" with Undo is shown.
4. Given a month with no selectable overflow days, when navigating months, then previous/next navigation updates the grid and disables navigation beyond the account's earliest allowed range.

## L2-018: Edit an existing entry
**Traces to:** L1-003

The user edits any recorded entry.

**Acceptance Criteria:**
1. Given an entry opened for edit, when the user changes the value and saves, then the entry updates for the owner and a toast "Entry updated — {value}" is shown.
2. Given an edit is saved, when persisted, then `updatedAt` changes and dependent computations (deltas, averages, badges, streaks) recompute.
3. Given a user attempts to edit an entry they do not own, when the request reaches the server, then it is rejected with `403`/`404` (L2-067).

## L2-019: Delete an entry with undo
**Traces to:** L1-003

The user deletes an entry, with a brief opportunity to undo.

**Acceptance Criteria:**
1. Given an entry, when the user deletes it, then it is removed and a toast "Entry deleted" with an Undo action is shown.
2. Given the delete toast, when Undo is activated within the toast window, then the entry is restored with its original date, value, and note.
3. Given the undo window elapses, when it expires, then the deletion is final and re-derived analytics exclude the entry.
4. Given a delete request, when it reaches the server, then ownership is enforced (L2-067).

## L2-020: Undo last logged weigh-in
**Traces to:** L1-003

Logging offers an immediate undo.

**Acceptance Criteria:**
1. Given a just-logged entry, when the user activates Undo on the confirmation toast, then that log operation is reverted to the entry's prior state (deleted if it was newly created, or restored to prior value if it was an update).
2. Given Undo reverts a create, when reverted, then no orphaned entry remains and analytics return to their prior values.

## L2-021: View goal progress
**Traces to:** L1-004

The goal screen shows progress toward the goal weight.

**Acceptance Criteria:**
1. Given start, current, and goal weights, when the goal screen renders, then percent complete = clamp(0..100) of (start − current) / (start − goal) is shown in the dial with the remaining amount ("{x} kg to go").
2. Given the screen, when it renders, then chips display Start, Now, and Goal values and a progress bar reflects the same percentage.
3. Given the user has reached or passed the goal, when the screen renders, then progress shows 100% and a completion state is presented rather than a negative remaining value.
4. Given viewport widths XS through XL, when the goal screen renders, then the dial and stat cards stack on XS and lay out in a multi-column arrangement on MD+ without clipping.

## L2-022: Edit goal
**Traces to:** L1-004

The user changes goal weight and/or target date.

**Acceptance Criteria:**
1. Given the edit-goal screen, when the user changes goal weight and/or target date and saves, then the Goal updates and all progress/pace figures recompute.
2. Given a target date not strictly in the future, when saved, then it is rejected.
3. Given a goal weight equal to the current weight, when saved, then it is rejected with guidance.

## L2-023: Pace projection & schedule status
**Traces to:** L1-004

The system projects a completion date and reports schedule status.

**Acceptance Criteria:**
1. Given at least two entries spanning ≥ 7 days, when pace is computed, then average change per week is derived from a smoothed recent trend (not two raw points) and a projected goal-completion date is shown.
2. Given the projected date is earlier than the target date, when displayed, then status reads "Ahead of schedule" with the number of days early; if later, "Behind schedule"; if within tolerance, "On track".
3. Given insufficient data (< 2 entries or < 7 days), when pace would be computed, then a neutral "Keep logging to see your pace" message is shown instead of a misleading projection.
4. Given the recent trend shows no movement or movement away from goal, when projected, then the UI communicates this honestly rather than showing an impossible date.

## L2-035: Streak tracking
**Traces to:** L1-007
